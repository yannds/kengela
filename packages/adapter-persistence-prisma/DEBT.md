# DEBT.md — @kengela/adapter-persistence-prisma

> Le port est un sas, pas une planque. Ce qui est enveloppe et faible figure ici.

| #   | Ce qui est enveloppe                   | Etat                       | Probleme                                                                                                                                                                                                                                                                                                                   | Cible de migration                                                                                        | Prio |
| --- | -------------------------------------- | -------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- | ---- |
| 1   | Prisma / Postgres                      | enveloppe via `PrismaLike` | On ne genere pas de client ici : on decrit une surface NARROW. Un vrai `PrismaClient` est cense etre structurellement compatible cote app, mais ce n'est PAS prouve par un test de compilation contre `@prisma/client`.                                                                                                    | Ajouter dans une app de composition un test-type qui affecte un `PrismaClient` genere a `PrismaLike`.     | P2   |
| 2   | Tests d'integration DB live            | ABSENT                     | Les tests unitaires passent par un fake `PrismaLike` en memoire : mapping, fail-closed, rotation/revocation sont couverts, mais aucune requete reelle Postgres (RLS, contraintes, JSON round-trip) n'est exercee. Ce n'est PAS de la dette de code : c'est un TIER de test a cabler.                                       | Job CI avec un Postgres ephemere + `prisma db push` du schema de reference + suite d'integration.         | P1   |
| 3   | Rotation transactionnelle              | conditionnelle             | `rotate` n'est atomique que si le client injecte expose `$transaction`. Sans lui, degradation en `delete` puis `create` sequentiels : une panne entre les deux peut laisser la session detruite sans remplacante.                                                                                                          | Exiger `$transaction` en prod (le vrai `PrismaClient` le fournit) ; documenter le contrat de composition. | P2   |
| 4   | Colonnes JSON (`ctx`, `obligations`)   | enveloppe                  | Stockees en `Json` opaque et relues via narrowing defensif fail-closed. Pas de schema JSON versionne : une derive de forme est toleree silencieusement (valeurs illisibles ecartees).                                                                                                                                      | Valider a l'ecriture (zod/valibot) + versionner la forme serialisee.                                      | P3   |
| 5   | `PrismaSubjectKeyStore.getOrCreateKey` | non atomique               | La creation de cle de sujet est un `findFirst` puis `create` : sous forte concurrence, deux appels simultanes pour un meme sujet inconnu peuvent creer deux lignes (course). Le chiffrement at-rest de la cle est en outre CONDITIONNEL a l'injection d'un `KeyManagementPort` (sans lui, cle en clair base64 = mode dev). | Upsert atomique sur contrainte unique `(tenantId, subjectId)` ; exiger un KMS injecte en prod.            | P3   |
