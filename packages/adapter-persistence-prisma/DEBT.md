# DEBT.md - @kengela/adapter-persistence-prisma

> The port is an airlock, not a hideout. What is wrapped and weak is listed here.

| #   | What is wrapped                        | State                    | Problem                                                                                                                                                                                                                                                                                        | Migration target                                                                                         | Prio |
| --- | -------------------------------------- | ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- | ---- |
| 1   | Prisma / Postgres                      | wrapped via `PrismaLike` | We do not generate a client here: we describe a NARROW surface. A real `PrismaClient` is supposed to be structurally compatible on the app side, but this is NOT proven by a compilation test against `@prisma/client`.                                                                        | Add, in a composition app, a type-test that assigns a generated `PrismaClient` to `PrismaLike`.          | P2   |
| 2   | Live DB integration tests              | ABSENT                   | Unit tests go through an in-memory `PrismaLike` fake: mapping, fail-closed, rotation/revocation are covered, but no real Postgres query (RLS, constraints, JSON round-trip) is exercised. This is NOT code debt: it is a test TIER to wire up.                                                 | CI job with an ephemeral Postgres + `prisma db push` of the reference schema + integration suite.        | P1   |
| 3   | Transactional rotation                 | conditional              | `rotate` is only atomic if the injected client exposes `$transaction`. Without it, it degrades to sequential `delete` then `create`: a failure between the two can leave the session destroyed with no replacement.                                                                            | Require `$transaction` in prod (the real `PrismaClient` provides it); document the composition contract. | P2   |
| 4   | JSON columns (`ctx`, `obligations`)    | wrapped                  | Stored as opaque `Json` and re-read via defensive fail-closed narrowing. No versioned JSON schema: a shape drift is silently tolerated (unreadable values discarded).                                                                                                                          | Validate on write (zod/valibot) + version the serialized shape.                                          | P3   |
| 5   | `PrismaSubjectKeyStore.getOrCreateKey` | non-atomic               | Subject key creation is a `findFirst` then `create`: under heavy concurrency, two simultaneous calls for the same unknown subject can create two rows (race). At-rest key encryption is moreover CONDITIONAL on injecting a `KeyManagementPort` (without it, cleartext base64 key = dev mode). | Atomic upsert on the unique constraint `(tenantId, subjectId)`; require an injected KMS in prod.         | P3   |
