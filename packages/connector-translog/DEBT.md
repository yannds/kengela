# DEBT.md - @kengela/connector-translog

> Le port est un sas, pas une planque. Ce qui est enveloppe, approxime ou perdu figure ici, honnetement.

Ce paquet vit en RÉFÉRENCE dans le monorepo Kengela : il prouve que les ports Kengela
s'implementent contre le schema Prisma REEL de TransLog Pro. Une fois `@kengela/*` publie,
ce connecteur est destine a etre DEPOSE dans TransLog (il y remplacera l'authn/authz maison).

| #   | Ce qui est enveloppe / limite                       | Etat                               | Probleme                                                                                                                                                                                                                                                                                                                                                                          | Cible de migration                                                                                                                     | Prio |
| --- | --------------------------------------------------- | ---------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- | ---- |
| 1   | `AuthContext` de session                            | LOSSY a l'ecriture ET a la lecture | Le schema `Session` de TransLog n'a que `ipAddress` + `userAgent`. Donc `geo`, `riskScore`, `device.id`, `device.trusted` sont PERDUS a la persistance, et `authTime` est approxime par `createdAt.getTime()`. La reconstitution ne peut jamais restituer ces signaux ZTNA.                                                                                                       | Ajouter une colonne `ctx Json` sur `Session` (ou une table dediee `SessionContext`) et brancher un `ContextProvider` cote TransLog.    | P1   |
| 2   | Mono-role                                           | structurel                         | TransLog est mono-role (`User.roleId` unique) : `CredentialRecord.roles` contient au plus un element, et `loadGrantsForUser` ne lit qu'un seul role. Le contrat Kengela est multi-role (union des grants) ; on n'exploite pas cette capacite.                                                                                                                                     | Quand TransLog migrera vers des affectations multi-roles (table de jointure User x Role), enrichir `roles` et l'agregation des grants. | P2   |
| 3   | Policies declaratives                               | ABSENTES                           | `TranslogPolicyStore.loadPolicies` retourne toujours `[]` : TransLog n'a pas (encore) de table de policies. L'autorisation repose sur le RBAC seul (grants issus de `RolePermission`). Aucune condition ABAC/CEL n'est chargee.                                                                                                                                                   | Introduire une table `Policy`/`PolicyRule` cote TransLog (+ overrides tenant) et implementer un vrai chargement.                       | P2   |
| 4   | `loadRole` sans table `Role` dans la surface NARROW | approximation                      | La surface NARROW n'expose PAS de delegue `role` (seulement `rolePermission`). `loadRole(roleKey)` derive donc l'existence du role de la PRESENCE de permissions : un role reel sans aucune permission est indistinguable d'un role absent (les deux -> `null`). `Role.name` n'est pas surface (le type Kengela `Role` n'en a pas besoin : il porte `key`, `tenantId`, `grants`). | Si besoin de distinguer role-vide vs role-absent, ajouter le delegue `role.findFirst` a la surface NARROW.                             | P3   |
| 5   | Compatibilite `PrismaClient` reel                   | enveloppe via `TranslogPrismaLike` | On decrit une surface NARROW, non prouvee par un test de compilation contre le `PrismaClient` genere de TransLog. Les lignes reelles sont des sur-ensembles de nos lignes ; l'assignabilite est presumee, pas verifiee ici.                                                                                                                                                       | Au depot dans TransLog : ajouter un test-type affectant le `PrismaClient` genere a `TranslogPrismaLike`.                               | P2   |
| 6   | Tests d'integration DB live                         | ABSENT                             | Les tests passent par un fake `TranslogPrismaLike` en memoire (mapping, fail-closed, rotation, isolation couverts) ; aucune requete Postgres reelle (RLS multi-tenant, contraintes, index unique `token`) n'est exercee.                                                                                                                                                          | Job CI avec Postgres ephemere + schema TransLog de reference + suite d'integration.                                                    | P1   |
| 7   | Rotation transactionnelle                           | conditionnelle                     | `rotate` n'est atomique que si le client injecte expose `$transaction`. Sinon degradation `delete` puis `create` : une panne entre les deux peut laisser la session detruite sans remplacante.                                                                                                                                                                                    | Exiger `$transaction` en prod (le vrai `PrismaClient` le fournit) ; documenter le contrat de composition.                              | P2   |
