# DEBT.md - @kengela/connector-translog

> The port is an airlock, not a hideout. What is wrapped, approximated or lost is listed here, honestly.

This package lives as a REFERENCE in the Kengela monorepo: it proves that the Kengela ports
implement against the REAL TransLog Pro Prisma schema. Once `@kengela/*` is published,
this connector is meant to be MOVED into TransLog (where it will replace the in-house authn/authz).

| #   | What is wrapped / limited                               | State                            | Problem                                                                                                                                                                                                                                                                                                                                                                              | Migration target                                                                                                                | Prio |
| --- | ------------------------------------------------------- | -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------- | ---- |
| 1   | Session `AuthContext`                                   | LOSSY on write AND on read       | The TransLog `Session` schema only has `ipAddress` + `userAgent`. So `geo`, `riskScore`, `device.id`, `device.trusted` are LOST on persistence, and `authTime` is approximated by `createdAt.getTime()`. The reconstitution can never restore these ZTNA signals.                                                                                                                    | Add a `ctx Json` column on `Session` (or a dedicated `SessionContext` table) and wire a `ContextProvider` on the TransLog side. | P1   |
| 2   | Single-role                                             | structural                       | TransLog is single-role (`User.roleId` unique): `CredentialRecord.roles` holds at most one element, and `loadGrantsForUser` reads a single role only. The Kengela contract is multi-role (union of grants); we do not exploit this capability.                                                                                                                                       | When TransLog migrates to multi-role assignments (User x Role join table), enrich `roles` and the grant aggregation.            | P2   |
| 3   | Declarative policies                                    | ABSENT                           | `TranslogPolicyStore.loadPolicies` always returns `[]`: TransLog has no policy table (yet). Authorization relies on RBAC alone (grants from `RolePermission`). No ABAC/CEL condition is loaded.                                                                                                                                                                                      | Introduce a `Policy`/`PolicyRule` table on the TransLog side (+ tenant overrides) and implement real loading.                   | P2   |
| 4   | `loadRole` without a `Role` table in the NARROW surface | approximation                    | The NARROW surface does NOT expose a `role` delegate (only `rolePermission`). `loadRole(roleKey)` therefore derives the role's existence from the PRESENCE of permissions: a real role with no permission is indistinguishable from an absent role (both -> `null`). `Role.name` is not surfaced (the Kengela `Role` type does not need it: it carries `key`, `tenantId`, `grants`). | If distinguishing empty-role vs absent-role is needed, add the `role.findFirst` delegate to the NARROW surface.                 | P3   |
| 5   | Real `PrismaClient` compatibility                       | wrapped via `TranslogPrismaLike` | We describe a NARROW surface, not proven by a compile-time test against TransLog's generated `PrismaClient`. The real rows are supersets of our rows; assignability is presumed, not verified here.                                                                                                                                                                                  | When moved into TransLog: add a type-test assigning the generated `PrismaClient` to `TranslogPrismaLike`.                       | P2   |
| 6   | Live DB integration tests                               | ABSENT                           | The tests go through an in-memory `TranslogPrismaLike` fake (mapping, fail-closed, rotation, isolation covered); no real Postgres query (multi-tenant RLS, constraints, unique `token` index) is exercised.                                                                                                                                                                          | CI job with an ephemeral Postgres + reference TransLog schema + integration suite.                                              | P1   |
| 7   | Transactional rotation                                  | conditional                      | `rotate` is atomic only if the injected client exposes `$transaction`. Otherwise it degrades to `delete` then `create`: a failure between the two can leave the session destroyed without a replacement.                                                                                                                                                                             | Require `$transaction` in prod (the real `PrismaClient` provides it); document the composition contract.                        | P2   |
